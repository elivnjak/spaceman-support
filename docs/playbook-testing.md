## Most Useful Commands

### Run one scenario against the live dev server

```bash
npm run playbook:test -- --scenario <scenario-id> --base-url http://127.0.0.1:3001
```

Example:

```bash
npm run playbook:test -- --scenario generated-resolution-too-runny-3080dfb7 --base-url http://127.0.0.1:3001
```

### Run a suite against the live dev server

```bash
npm run playbook:test -- --suite generated-resolution --base-url http://127.0.0.1:3001
```

### Run with draft fix generation

```bash
npm run playbook:test -- --scenario <scenario-id> --base-url http://127.0.0.1:3001 --fix
```

### Regenerate all autogenerated scenarios

```bash
npm run playbook:test:generate
```

### Run harness unit tests

```bash
npm run test:playbook-harness
```

## What The Test Runner Mutates

Running the live harness against `--base-url` does not automatically rewrite playbooks.

It creates normal runtime data only, such as:

- diagnostic sessions
- audit logs
- uploaded test images
- normal per-session chat state

Actual playbook/action changes are done separately through the repo's Drizzle DB layer, usually via:

- [migrate-playbooks-to-v2.ts](/Users/elivnjak/Sites/ai-rag-saas/scripts/migrate-playbooks-to-v2.ts)
- one-off `node --import tsx` DB update snippets

After live DB updates, workbook snapshots can be exported with:

- [export-structured-playbooks.ts](/Users/elivnjak/Sites/ai-rag-saas/scripts/export-structured-playbooks.ts)

## Scenario Authoring Notes

Each scenario directory should contain:

- `scenario.json`
- optional `fixtures/`

Images must be scenario-local. Put them under that scenario’s `fixtures/` folder and reference them from the scenario.

The schema supports:

- explicit user turns
- `autoRespond` turns
- fixture-backed images
- turn-by-turn expectations
- final expectations

Useful expectation fields:

- turn-level `expect.resolutionStepIds` for asserting a specific resolving turn
- final `finalExpect.resolutionStepIds` for asserting the final authored step set when a scenario may resolve early

Good uses for scenarios:

- known bugs
- high-value playbooks
- edge-case escalation logic
- image-dependent flows
- diagnosis contradiction checks

## Important Workflow Rules

### 1. Use `--base-url` for live local runs

For day-to-day testing, the most reliable workflow is the live local dev server:

```bash
npm run dev
```

Then run:

```bash
npm run playbook:test -- --scenario <scenario-id> --base-url http://127.0.0.1:3001
```

### 2. Diagnosis mode is bypassed in harness mode

The chat route forces diagnosis on for harness requests with:

- `x-playbook-test-mode: true`

That means the test harness can still run full diagnosis workflows even if diagnosis mode is disabled for normal users.

### 3. Scenarios can get stuck if skip replies are identical

The live chat route deduplicates near-identical repeated user turns over a short window.

So if multiple auto-response turns all say exactly:

- `I don't have a photo.`

the test may replay the same assistant reply instead of progressing.

Prefer unique skip messages per request type, for example:

- `I don't have a product photo.`
- `I don't have a display photo.`

See the regression scenario here:

- [scenario.json](/Users/elivnjak/Sites/ai-rag-saas/data/playbook_tests/regression-ss-soft-runny-no-supported-cause/scenario.json)

## Diagnosis Quality Checks

The most important rule is:

- the system should only resolve when the collected evidence supports the cause
- otherwise it should keep gathering evidence or escalate

The final diagnosis verifier and step-selection logic live in:

- [src/lib/pipeline/diagnostic-planner.ts](/Users/elivnjak/Sites/ai-rag-saas/src/lib/pipeline/diagnostic-planner.ts)
- [src/app/api/chat/route.ts](/Users/elivnjak/Sites/ai-rag-saas/src/app/api/chat/route.ts)

The current split is:

- LLM verifier decides whether the proposed cause is actually supported by the collected evidence and which authored playbook steps still apply
- code enforces structure, grounding, allowed step IDs, and escalation plumbing

## Recommended Future Workflow

When fixing playbooks or diagnosis logic:

1. Run one failing scenario against the live dev server.
2. Inspect the scenario JSON report and audit log.
3. Patch code, playbook content, or scenario data.
4. Rerun the same scenario.
5. Rerun a wider suite, usually `generated-resolution`.
6. If considering a model change, use the replay comparator first before switching defaults.

## Current Limitations

- The broad live suite can take a while to finish.
- The `intent_manifest` DB override table may not exist locally, so model experiments may need to be done by changing defaults instead of DB overrides.
- The replay comparison scripts compare planner prompts directly, which is faster than full end-to-end A/B, but it is still not a perfect substitute for a full live suite rerun.
