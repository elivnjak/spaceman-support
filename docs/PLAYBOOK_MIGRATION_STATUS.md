## How Live DB Updates Are Being Done

There are two separate workflows:

1. running tests against the live dev server
2. updating playbooks/actions in the live DB

They are not the same thing.

### What the live test runner does

When running:

```bash
npm run playbook:test -- --scenario <scenario-id> --base-url http://127.0.0.1:3001
```

the harness exercises the real `/api/chat` route and creates normal runtime data such as:

- diagnostic sessions
- audit logs
- uploaded test images
- normal per-session chat state

It does not automatically rewrite playbook definitions.

### What actually updates playbooks/actions

Playbook and action changes have been applied directly to the local Postgres DB through the repo's Drizzle client in:

- [src/lib/db/index.ts](/Users/elivnjak/Sites/ai-rag-saas/src/lib/db/index.ts)

The two main update paths are:

- repo scripts such as [migrate-playbooks-to-v2.ts](/Users/elivnjak/Sites/ai-rag-saas/scripts/migrate-playbooks-to-v2.ts)
- one-off `node --import tsx` snippets that import `db`, `playbooks`, and `actions`

Those updates write directly to the JSONB-backed DB fields, typically:

- `schemaVersion`
- `evidenceChecklist`
- `candidateCauses`
- `updatedAt`
- and sometimes action `expectedInput` or evidence `actionId` wiring

So when this migration says the "live DB was updated", it means a deliberate DB update was run through Drizzle, not that the test harness silently changed playbooks.

## Safe Single-Playbook Update Template

This is the reusable pattern for updating one live playbook directly in the DB.

```bash
node --import tsx - <<'NODE'
import DB from './src/lib/db/index.ts';
import { eq } from 'drizzle-orm';

const { db, playbooks } = DB;
const playbookId = '<playbook-id>';

const [row] = await db.select().from(playbooks).where(eq(playbooks.id, playbookId)).limit(1);
if (!row) throw new Error(`Playbook not found: ${playbookId}`);

const evidenceChecklist = Array.isArray(row.evidenceChecklist) ? [...row.evidenceChecklist] : [];
const candidateCauses = Array.isArray(row.candidateCauses) ? [...row.candidateCauses] : [];

// Modify evidenceChecklist and candidateCauses here.
// Example:
// const idx = candidateCauses.findIndex((cause) => cause.id === 'cause_example');
// candidateCauses[idx] = {
//   ...candidateCauses[idx],
//   supportMode: 'all',
//   supportRules: [
//     { evidenceId: 'ev_example', operator: 'equals', value: 'Yes' },
//   ],
//   excludeRules: [
//     { evidenceId: 'ev_other', operator: 'equals', value: 'No' },
//   ],
// };

await db
  .update(playbooks)
  .set({
    schemaVersion: 2,
    evidenceChecklist,
    candidateCauses,
    updatedAt: new Date(),
  })
  .where(eq(playbooks.id, playbookId));

console.log(`Updated playbook ${playbookId}`);
NODE
```

If an evidence item needs real enum input instead of a generic yes/no prompt, update or create the matching action row in the same way, then wire the evidence item via `actionId`.

## SaaS-Grade Rules

The current direction is:

1. Playbooks own domain semantics.
2. Code owns workflow, state, validation, and orchestration.
3. LLMs handle extraction, clarification, and user-facing explanation.

Do:

- encode diagnosis semantics in structured playbook data
- use generic runtime evaluation over `supportRules` / `excludeRules`
- wire evidence items to real actions when request shape matters
- keep live runs sequential to avoid rate limits and process buildup

Do not:

- hardcode product thresholds in app code
- add playbook-ID-specific diagnosis branches
- use cause-name keyword heuristics as the source of truth
- patch only one cause if sibling causes overlap semantically

## Key Findings So Far

- Prose-only causes are the main source of wrong diagnoses and unnecessary escalations.
- Missing `actionId` wiring causes the live app to fall back to generic yes/no prompts even when the evidence is really enum-based.
- Repeated-request cleanup is often benign once the correct structured cause is already supported.
- Generated scenarios must be regenerated after playbook or action updates or they can become stale and misleading.
- The live local DB is currently ahead of some workbook snapshots in the repo, so exports need periodic catch-up.

## Exact Workflow Used For Testing And Fixing

This is the full practical workflow that has been used for the migration so far.

It is intentionally repetitive. The consistency is what makes it reliable.

### Phase 0. Start Conditions

Before doing anything:

1. Make sure the local app is running:

```bash
npm run dev
```

2. Make sure local Postgres is reachable through `DATABASE_URL`.
3. Assume the live local DB is the current source of truth for playbook semantics during the migration.
4. Avoid parallel live runs. Use one scenario at a time.

### Phase 1. Choose The Next Family

The family is chosen from the backlog in this file, usually by:

- highest recent failure frequency
- strongest semantic overlap between sibling causes
- proximity to already-converted families
- whether the failure looks like a playbook problem rather than a generic runtime bug

The important rule is:

- convert whole families, not isolated causes

If one cause in a family fails, inspect the entire sibling set before making any semantic changes.

### Phase 2. Inspect The Current Live Playbook

Before running anything, inspect the current DB row:

```bash
node --import tsx - <<'NODE'
import DB from './src/lib/db/index.ts';
import { eq } from 'drizzle-orm';
const { db, playbooks } = DB;
const id = '<playbook-id>';
const [pb] = await db.select().from(playbooks).where(eq(playbooks.id, id)).limit(1);
console.log(JSON.stringify(pb, null, 2));
NODE
```

While inspecting, explicitly look for:

- candidate causes with no `supportRules`
- sibling causes with overlapping prose semantics
- evidence items that imply enum comparison but have no `actionId`
- evidence items whose wording is biased or polarity-confused
- causes that should end in escalation instead of operator resolution

If action wiring may be involved, inspect the relevant action rows too.

### Phase 3. Run One Live Generated Scenario First

Pick one generated cause-resolution scenario for the family and run it live:

```bash
npm run playbook:test -- --scenario <scenario-id> --base-url http://127.0.0.1:3001
```

Do not run sibling scenarios yet.

The first run is used to classify the failure, not to prove the whole family.

### Phase 4. Read The Failure Artifact

After the run finishes, inspect:

- `logs/playbook-tests/<run-id>/summary.md`
- `logs/playbook-tests/<run-id>/scenarios/<scenario-id>.json`

When reading the scenario artifact, check:

- what evidence was actually collected
- which requests were asked and in what order
- whether any requests were removed by dedupe
- what cause the planner proposed
- what cause the verifier accepted or rejected
- whether the final phase was `gathering_info`, `diagnosing`, `resolving`, or `escalated`

The main question at this stage is:

- is the failure caused by playbook semantics, runtime plumbing, or stale/generated scenario data?

### Phase 5. Classify The Failure

Use this decision rule.

If the live app asked the wrong input shape:

- the problem is usually action wiring
- fix the evidence `actionId` or the action `expectedInput`

If the live app collected good evidence but resolved to the wrong sibling cause:

- the problem is usually playbook semantics
- convert the whole family with structured `supportRules` / `excludeRules`

If the live app collected the right evidence but stalled, repeated, or dead-ended:

- inspect generic runtime handling
- common places:
  - [request-inference.ts](/Users/elivnjak/Sites/ai-rag-saas/src/lib/pipeline/request-inference.ts)
  - [request-dedup.ts](/Users/elivnjak/Sites/ai-rag-saas/src/lib/pipeline/request-dedup.ts)
  - [diagnostic-planner.ts](/Users/elivnjak/Sites/ai-rag-saas/src/lib/pipeline/diagnostic-planner.ts)
  - [route.ts](/Users/elivnjak/Sites/ai-rag-saas/src/app/api/chat/route.ts)

If the failure only appears after a recent playbook/action change:

- regenerate scenarios first
- stale scenario data is a common false failure source

### Phase 6. Fix The Whole Family In The Live DB

Once the root cause is understood, update the live DB directly through Drizzle.

The usual change pattern is:

1. copy the current `evidenceChecklist`
2. copy the current `candidateCauses`
3. update evidence wiring and value semantics
4. add `supportRules` / `excludeRules` for all sibling causes
5. set `supportMode`
6. add `outcome: "escalation"` where needed
7. save the playbook row

If the family needs a new or corrected action:

1. inspect or create the action row
2. set the correct `expectedInput`
3. wire the evidence item with `actionId`
4. only then rerun generation/tests

This is usually done through a one-off `node --import tsx` snippet, not through the harness.

### Phase 7. Apply Generic Runtime Fixes Only If The Problem Is Truly Generic

Do not patch runtime code just because one playbook failed.

Only change app code if the defect is clearly generic, for example:

- enum answers from live requests are not being extracted reliably
- skipped/uncertain evidence blocks future real requests
- structured escalation causes are supported but the runtime still dead-ends
- benign sanitization noise is causing false harness failures

When a code fix is made:

1. keep it generic
2. avoid playbook IDs and product-specific thresholds
3. add a focused regression test in the harness/test layer

### Phase 8. Regenerate Scenarios Immediately

After any playbook or action change:

```bash
npm run playbook:test:generate
```

Do not skip this.

The generated pack often becomes stale as soon as:

- request options change
- an evidence item gets wired to a real action
- structured cause rules change the expected evidence path

### Phase 9. Re-Run The Entire Family Sequentially

Once generation is complete, run every generated cause-resolution scenario in that family, one by one.

This is important for two reasons:

1. sibling causes often overlap semantically
2. passing the original failing branch does not prove the family is stable

The actual working pattern used during migration is:

1. rerun the original failing branch
2. rerun every sibling branch in that family
3. if one sibling now regresses, refine the family semantics again
4. repeat until the entire family is green

### Phase 10. Re-Run Local Verification

After the family is green live, rerun:

```bash
npm run test:playbook-harness
npx tsc --noEmit
```

This catches:

- harness expectation regressions
- prompt/sanitizer regressions
- type drift from runtime changes

### Phase 11. Export Workbook Snapshots

Once a family or batch is stable, export workbook snapshots so the repo catches up with the live DB:

```bash
node --import tsx scripts/export-structured-playbooks.ts
```

Exports go to:

- [data/playbook_workbooks](/Users/elivnjak/Sites/ai-rag-saas/data/playbook_workbooks)

This step has sometimes lagged behind the live DB during the migration, so check whether the exported workbook actually reflects the latest DB semantics.

### Phase 12. Update The Handoff

After each stable batch:

1. update this migration doc
2. update the backlog counts if needed
3. note any new generic runtime fix
4. note any newly added action wiring pattern
5. record any family-specific gotcha that would help the next pass

## Common Failure Patterns

### Prose-only causes collapse into the wrong sibling

Symptom:

- the planner picks a plausible but wrong sibling cause

Fix:

- add full-family structured `supportRules` / `excludeRules`

### Evidence has enum semantics but no action wiring

Symptom:

- live app asks generic yes/no
- scenario answers stop matching the live request

Fix:

- create or reuse an action with the correct `expectedInput`
- wire the evidence item to that action

### Repeated-request cleanup after the final evidence arrives

Symptom:
